## نبذة عن المشروع

هذا المشروع عبارة عن موقع HTML مستقل تم تصميمه لمساعدة الشركة على جمع بيانات الموظفين وتجهيزها قبل رفعها أو ترحيلها إلى نظام **ZenHR**.

الموقع لا يحتاج إلى React أو Node.js أو Backend تقليدي. هو عبارة عن ملف `index.html` يعمل مباشرة من المتصفح، ويرتبط مع **Firebase Firestore** لتخزين بيانات الموظفين والنتائج والتعديلات.

تم تصميم التجربة لتكون بسيطة وواضحة للموظف، وفي نفس الوقت قوية للإدارة، بحيث يمكن للأدمن متابعة من قام بتعبئة البيانات، ومن لم يقم بذلك، وتنزيل النتائج بملفات منفصلة للموظفين والعائلة والعهد.

---

## الهدف من الموقع

الهدف الأساسي هو تجهيز قاعدة بيانات منظمة ودقيقة للموظفين قبل إدخالها إلى ZenHR.

الموقع يساعد في:

- جمع البيانات الشخصية الأساسية.
- جمع بيانات الهوية والجواز والإقامة لغير الأردنيين.
- جمع معلومات الاتصال.
- جمع معلومات العمل والعقد.
- جمع معلومات البنك والآيبان.
- جمع بيانات العهد التي بحوزة الموظف.
- جمع بيانات أفراد العائلة عند الحاجة.
- مقارنة البيانات الأصلية مع ما عدّله الموظف.
- استخراج ملفات قابلة للفتح في Excel.
- معرفة الموظفين الذين لم يعبئوا بياناتهم بعد.

## المشكلة التي يحلها الموقع

عند الانتقال إلى نظام HR جديد مثل ZenHR، تكون بيانات الموظفين عادة موزعة بين أكثر من مصدر:

- ملفات ضمان اجتماعي.
- ملفات عقود.
- ملفات Survey قديمة.
- ملف إيميلات.
- ملفات رواتب.
- معلومات ناقصة عند HR.

والمشكلة أن إدخال هذه البيانات يدويًا يسبب:

- أخطاء في الأسماء.
- أخطاء في الأرقام الوطنية أو أرقام الجوازات.
- نقص في الحقول المطلوبة.
- صعوبة في معرفة من أكمل ومن لم يكمل.
- صعوبة في تجهيز أكثر من ملف Import.
- عدم وجود سجل واضح للتعديلات.

هذا الموقع يحل هذه المشكلة عبر تجربة منظمة:

1. تحميل قاعدة موظفين أولية.
2. دخول الموظف بطريقة سهلة.
3. عرض بياناته الموجودة مسبقًا.
4. السماح له بتصحيح أو استكمال البيانات.
5. حفظ كل شيء في Firestore.
6. تمكين الأدمن من متابعة وتنزيل النتائج.

## الفكرة العامة لتجربة الموظف

الموظف لا يرى تعقيدات Firebase أو ملفات Excel أو ZenHR.

هو يرى واجهة بسيطة تشبه نموذج Typeform، ولكن بتجربة أكثر تخصيصًا.

خطوات الموظف:

1. يفتح رابط الموقع.
2. يدخل تاريخ ميلاده.
3. يدخل أول حرف من اسمه باللغة الإنجليزية.
4. يضغط على زر عرض بياناته.
5. تظهر له بياناته الموجودة مسبقًا.
6. يراجع البيانات.
7. يكمل الأسئلة المطلوبة.
8. يختار العهد الموجودة لديه أو يختار أنه لا يملك عهدة.
9. إذا كان متزوجًا أو مطلقًا أو أرملًا، تظهر له أسئلة أفراد العائلة.
10. يراجع الملخص النهائي.
11. يرسل البيانات.

بعد الإرسال، يتم حفظ النتيجة في Firestore، وتظهر مباشرة في لوحة الأدمن.

---

## الفكرة العامة لتجربة الأدمن

الأدمن لديه لوحة تحكم مخفية داخل نفس الموقع.

طريقة الوصول:

- يتم عمل double click على علامة HR في الواجهة.
- تظهر شاشة دخول الأدمن.
- الدخول الحقيقي يتم من خلال Firebase Authentication باستخدام Email/Password.
- بعد الدخول تظهر لوحة التحكم.

لوحة الأدمن توفر:

- إجمالي عدد الموظفين في قاعدة البيانات.
- عدد الموظفين الذين أرسلوا البيانات.
- عدد الموظفين الذين لم يرسلوا البيانات.
- عدد التعديلات التي قام بها الموظفون.
- جدول النتائج.
- جدول الموظفين الذين لم يعبئوا البيانات.
- جدول التعديلات قبل وبعد.
- جدول قاعدة الموظفين المستوردة.
- أزرار تنزيل ملفات الموظفين والعائلة والعهد.
- زر Import Seed لإدخال بيانات الموظفين إلى Firestore.
- زر تهيئة الأدمن والـ collections في أول تشغيل.

## الملفات الموجودة في المشروع

عادة تحتوي النسخة على الملفات التالية:

```txt
index.html
README.md
firestore.rules
.gitignore
private-tools/
```

وفي النسخة الكاملة المحلية قد يوجد أيضًا:

```txt
private-tools/employee-seed-private.json
```

### `index.html`

هو ملف الموقع الرئيسي. يحتوي على:

- الواجهة.
- CSS.
- JavaScript.
- Firebase config.
- منطق تسجيل الدخول.
- منطق حفظ البيانات.
- منطق لوحة الأدمن.
- منطق التصدير.
- منطق الاستيراد.

### `README.md`

هذا الملف. يحتوي على شرح المشروع وطريقة تشغيله وإدارته.

### `firestore.rules`

يحتوي قواعد الأمان الخاصة بـ Firestore.

هذه القواعد مهمة جدًا لأن الموقع سيتم رفعه على GitHub، وبالتالي كود الواجهة سيكون متاحًا للمتصفح. الحماية الحقيقية ليست بإخفاء زر الأدمن، بل بقواعد Firestore وFirebase Authentication.

### `.gitignore`

ملف يمنع رفع الملفات الحساسة إلى GitHub عند استخدام Git بشكل صحيح.

### `private-tools/employee-seed-private.json`

ملف بيانات الموظفين الأولية.

هذا الملف حساس ولا يجب رفعه إلى GitHub public.

## الفرق بين النسخة الآمنة ونسخة التشغيل المحلي

يوجد عادة إصداران من المشروع:

### 1. GitHub Safe Version

هذه النسخة مخصصة للرفع على GitHub Pages.

يجب أن تحتوي فقط على:

```txt
index.html
README.md
firestore.rules
.gitignore
```

ولا يجب أن تحتوي على بيانات الموظفين.

### 2. Full Local Setup Version

هذه النسخة تستخدم داخليًا من قبل الأدمن فقط.

قد تحتوي على:

```txt
private-tools/employee-seed-private.json
```

الغرض منها هو استيراد بيانات الموظفين إلى Firebase مرة واحدة.

بعد الاستيراد، لا تحتاج أن تنشر ملف seed.

## المتطلبات قبل التشغيل

تحتاج إلى:

1. مشروع Firebase جاهز.
2. Firebase Authentication مفعّل.
3. Firestore Database مفعّل.
4. Firestore Security Rules منشورة.
5. حساب أدمن داخل Firebase Authentication.
6. رابط GitHub Pages مضاف إلى Authorized Domains.
7. عدم نشر ملفات الموظفين الحساسة على GitHub.

## إعداد Firebase خطوة بخطوة

من Firebase Console:

1. افتح مشروع Firebase.
2. انتقل إلى **Build**.
3. فعّل **Authentication**.
4. فعّل **Firestore Database**.
5. انسخ قواعد الأمان من ملف `firestore.rules`.
6. أضف دومين GitHub Pages في Authorized Domains.
7. أنشئ حساب الأدمن.
8. افتح الموقع.
9. ادخل إلى لوحة الأدمن.
10. اضغط تهيئة الأدمن.
11. استورد ملف seed.

---

## تفعيل Authentication

من Firebase Console:

```txt
Build → Authentication → Get started
```

ثم من تبويب Sign-in method فعّل:

### Anonymous

يستخدم للموظفين.

الموظف لا يحتاج أن يكون لديه حساب Firebase. عند فتح الموقع، يقوم التطبيق بتسجيل دخوله بشكل مجهول حتى يستطيع تنفيذ عمليات القراءة والكتابة المسموحة في Firestore Rules.

### Email/Password

يستخدم للأدمن.

الأدمن يجب أن يدخل من شاشة الأدمن المخفية باستخدام بريد وكلمة مرور تم إنشاؤهما داخل Firebase Authentication.

---

## تفعيل Firestore Database

من Firebase Console:

```txt
Build → Firestore Database → Create database
```

اختر:

```txt
Production mode
```

لا تستخدم Test Mode في التشغيل الحقيقي لأن Test Mode قد يفتح قاعدة البيانات بشكل خطير إذا لم يتم تعديل القواعد.

بعد إنشاء قاعدة البيانات، انتقل إلى Rules وانشر القواعد الموجودة في ملف `firestore.rules`.

---

## نشر Firestore Security Rules

افتح:

```txt
Firestore Database → Rules
```

ثم الصق محتوى ملف:

```txt
firestore.rules
```

واضغط:

```txt
Publish
```

القواعد تضبط الصلاحيات كالتالي:

- الموظف يستطيع الوصول فقط للعمليات المطلوبة لتعبئة بياناته.
- الأدمن يستطيع قراءة النتائج وقاعدة الموظفين والتصدير.
- إنشاء وثيقة الأدمن الأولى يتم عن طريق حساب Bootstrap Admin المحدد مسبقًا.
- لا يمكن لأي شخص عادي قراءة كل الموظفين.
- لا يمكن لأي شخص عادي تنزيل كل النتائج.

## تهيئة الأدمن لأول مرة

بعد نشر القواعد وتشغيل الموقع:

1. افتح الموقع.
2. اعمل double click على علامة HR.
3. أدخل بريد وكلمة مرور الأدمن التي أنشأتها في Firebase Authentication.
4. إذا كان هذا هو حساب الأدمن الصحيح، سيظهر زر:

```txt
تهيئة الأدمن والـ collections
```

5. اضغط الزر مرة واحدة.

سيقوم الموقع بإنشاء:

```txt
admins/{adminUid}
meta/setup
```

ولا تحتاج أن تنشئ collections يدويًا.

ملاحظة: Firestore لا يحتاج إنشاء collection فارغ يدويًا. الـ collection تظهر تلقائيًا عند إنشاء أول document داخلها.

## استيراد بيانات الموظفين إلى Firestore

بعد دخول الأدمن وتهيئة الحساب:

1. افتح لوحة الأدمن.
2. اضغط زر Import Seed.
3. اختر ملف:

```txt
employee-seed-private.json
```

4. سيقوم الموقع بإنشاء سجلات الموظفين داخل:

```txt
employeeDirectory
```

5. وسيقوم بإنشاء مفاتيح الدخول داخل:

```txt
loginIndex
```

كل موظف يتم حفظه باستخدام Temporary Employment Number مثل:

```txt
OWIP-0001
OWFS-0001
INF-0001
BAI-0001
```

هذه الأرقام مؤقتة إلى حين اعتماد الرقم الوظيفي الرسمي من ZenHR أو من الشركة.

## رفع الموقع على GitHub Pages

بعد التأكد أن البيانات موجودة في Firestore، ارفع فقط النسخة الآمنة.

الملفات التي يمكن رفعها:

```txt
index.html
README.md
firestore.rules
.gitignore
```

لا ترفع:

```txt
employee-seed-private.json
private-tools/
ملفات Excel الأصلية
أي ملف يحتوي أرقام وطنية
أي ملف يحتوي تواريخ ميلاد
أي ملف يحتوي رواتب
أي ملف يحتوي أرقام ضمان اجتماعي
```

بعد رفع الملفات:

1. افتح Repository على GitHub.
2. ادخل إلى Settings.
3. ادخل إلى Pages.
4. اختر branch المناسب، غالبًا `main`.
5. احفظ.
6. انسخ رابط الموقع.
7. أضف الرابط داخل Firebase Authorized Domains.

## شرح واجهة الموظف

واجهة الموظف مقسمة إلى مراحل واضحة:

### 1. الدخول

الموظف يدخل:

- تاريخ الميلاد.
- أول حرف من اسمه باللغة الإنجليزية.

مثال:

```txt
Birthdate: 1994-06-20
Initial: A
```

إذا تم العثور على موظف واحد مطابق، يتم نقله مباشرة إلى شاشة المراجعة.

إذا تم العثور على أكثر من موظف بسبب تشابه تاريخ الميلاد وأول حرف الاسم، يمكن إظهار خيارات للمطابقة حسب البيانات المتاحة.

### 2. مراجعة البيانات

تظهر بيانات الموظف الموجودة مسبقًا.

الغرض من هذه الشاشة أن يتأكد الموظف أن البيانات تخصه.

### 3. الأسماء

يتم إدخال أو مراجعة:

- First Name En
- Second Name En
- Third Name En
- Last Name En
- First Name Ar
- Second Name Ar
- Third Name Ar
- Last Name Ar

### 4. الهوية والجواز

يتم إدخال:

- Passport Number
- Passport Issue Date
- Passport Expiry Date
- Labor Card ID لغير الأردنيين
- Labor Card Expiry لغير الأردنيين
- Resident ID لغير الأردنيين
- MOL Number لغير الأردنيين
- National ID
- Nationality
- Religion

### 5. التواصل

يتم إدخال:

- Work Email
- Personal Email
- Mobile Number
- Phone Number
- Extension Number
- Address

### 6. الوظيفة والعقد

يتم إدخال أو مراجعة:

- Hiring Date
- End of Probation Date
- Contract Type
- Contract Duration
- End of Contract Date
- Position in English
- Position in Arabic
- Work Location
- Department
- Section
- Work Type

### 7. البنك

يتم إدخال:

- Bank Name
- Bank Branch
- Account Number
- IBAN Number

### 8. العهد

الموظف يختار أحد الخيارين:

- ليس لدي عهدة.
- لدي عهدة.

إذا اختار لديه عهدة، تظهر قائمة checklist بالعهد.

### 9. العائلة

تظهر فقط إذا كانت الحالة الاجتماعية:

- Married
- Divorced
- Widowed

ولا تظهر إذا كانت:

- Single

### 10. الإرسال النهائي

الموظف يراجع الملخص ويرسل البيانات.

## شرح واجهة الأدمن

لوحة الأدمن مقسمة إلى Tabs.

### النتائج

تعرض الموظفين الذين قاموا بإرسال البيانات.

تحتوي عادة على:

- Employee ID
- Name
- Company
- Submitted At
- Changes Count

### لم يعبئوا

تعرض الموظفين الموجودين في قاعدة الموظفين لكن لم يرسلوا البيانات بعد.

هذه القائمة مفيدة للتذكير.

### التعديلات

تعرض ما قام الموظف بتغييره مقارنة بالبيانات الأصلية.

مثال:

| Employee ID | Field | Before | After |
|------------|-------|--------|-------|
| OWIP-0001 | Work Email | old@email.com | new@email.com |

### قاعدة الموظفين

تعرض البيانات الأولية المستوردة من seed.

### أزرار التصدير

الأدمن يستطيع تنزيل:

- ملف الموظفين.
- ملف العائلة.
- ملف العهد.

## بنية البيانات داخل Firestore

الموقع يستخدم عدة Collections.

### `admins`

تحتوي على حسابات الأدمن.

مثال document:

```json
{
  "role": "admin",
  "active": true,
  "createdAt": "2026-06-16T00:00:00.000Z"
}
```

### `employeeDirectory`

تحتوي على قاعدة الموظفين الأولية.

كل موظف document مستقل.

مثال:

```json
{
  "employmentNumber": "OWIP-0001",
  "fullNameEn": "Example Employee",
  "fullNameAr": "مثال موظف",
  "company": "OWIP",
  "workEmail": "employee@example.com",
  "birthdate": "1994-01-01",
  "birthdateKey": "1994-01-01",
  "firstInitialKey": "E"
}
```

### `loginIndex`

يساعد على تسجيل دخول الموظف بطريقة تاريخ الميلاد + أول حرف.

المفتاح يكون عادة:

```txt
1994-01-01_E
```

ويحتوي على قائمة employeeIds المطابقة.

مثال:

```json
{
  "employeeIds": ["OWIP-0001"],
  "updatedAt": "2026-06-16T00:00:00.000Z"
}
```

### `submissions`

تحتوي على إجابات الموظفين النهائية.

مثال:

```json
{
  "employeeId": "OWIP-0001",
  "temporaryEmploymentNumber": "OWIP-0001",
  "status": "submitted",
  "employee": {},
  "familyMembers": [],
  "assets": [],
  "changeSummary": [],
  "submittedAt": "2026-06-16T00:00:00.000Z"
}
```

### `editLogs`

تحتوي على تفاصيل التعديلات.

مثال:

```json
{
  "employeeId": "OWIP-0001",
  "field": "workEmail",
  "before": "old@example.com",
  "after": "new@example.com",
  "changedAt": "2026-06-16T00:00:00.000Z"
}
```

### `meta`

تستخدم لحفظ بيانات النظام العامة، مثل وقت التهيئة وعدد عمليات الاستيراد.

## استكشاف الأخطاء الشائعة

### المشكلة: لوحة الأدمن لا تفتح

تأكد أنك تعمل double click على علامة HR وليس click واحد.

### المشكلة: لا أستطيع الدخول كأدمن

تحقق من:

- Email/Password مفعّل في Firebase Authentication.
- حساب الأدمن موجود.
- كلمة المرور صحيحة.
- UID الموجود في الكود والقواعد هو نفس UID الخاص بالأدمن.
- وثيقة الأدمن تم إنشاؤها عبر زر التهيئة.

### المشكلة: يظهر Permission Denied

الأسباب المحتملة:

- لم تنشر Firestore Rules الصحيحة.
- لم يتم تفعيل Anonymous Auth.
- لم يتم تفعيل Email/Password Auth.
- المستخدم ليس أدمن.
- الدومين غير مضاف إلى Authorized Domains.

### المشكلة: الموظف لا يظهر عند الدخول

تحقق من:

- تم استيراد seed إلى Firestore.
- تاريخ الميلاد في seed مطابق لصيغة yyyy-mm-dd.
- أول حرف الاسم الإنجليزي صحيح.
- يوجد document داخل loginIndex.

### المشكلة: يوجد أكثر من موظف بنفس تاريخ الميلاد ونفس الحرف

هذا ممكن لأن طريقة الدخول مبسطة.

في هذه الحالة يجب أن يتعامل النظام مع أكثر من نتيجة، أو يمكن مستقبلًا إضافة تحقق ثالث مثل آخر 4 أرقام من الهاتف أو الرقم الوطني.

### المشكلة: ملفات التصدير لا تفتح كـ Excel

الملفات بصيغة CSV متوافقة مع Excel.

افتح Excel ثم:

```txt
Data → From Text/CSV
```

أو افتح الملف مباشرة ثم احفظه بصيغة XLSX.

### المشكلة: تم رفع seed إلى GitHub بالخطأ

الإجراء المقترح:

1. احذف الملف فورًا.
2. اجعل repository private مؤقتًا.
3. غيّر قواعد Firestore إذا لزم.
4. اعتبر أن البيانات قد تكون انكشفت إذا كان repository public.
5. استخدم أدوات تنظيف Git history إذا احتجت حذف الملف من تاريخ Git.

### لماذا الموقع HTML واحد؟

للتجربة الأولية أو عند وجود مشكلة مؤقتة في Firebase، لكن التشغيل الحقيقي يجب أن يكون عبر Firestore.

لأن بعض الموظفين قد لا يكون لديهم بريد عمل أو قد يكون البريد في الملف غير مكتمل.

### لماذا قسم العائلة لا يظهر للجميع؟

لتحسين تجربة المستخدم وتقليل الأسئلة غير الضرورية.

## ملخص سريع للتشغيل

1. فعّل Firebase Authentication:
   - Anonymous
   - Email/Password

2. فعّل Firestore Database بوضع Production.

3. انشر قواعد:

```txt
firestore.rules
```

4. أضف دومين GitHub Pages إلى Authorized Domains.

5. افتح الموقع.

6. Double click على HR.

7. سجّل دخول الأدمن.

8. اضغط تهيئة الأدمن والـ collections.

9. استورد ملف seed الخاص.

10. ارفع النسخة الآمنة فقط على GitHub.

11. أرسل الرابط للموظفين.

12. تابع النتائج من لوحة الأدمن.

13. نزّل ملفات CSV المتوافقة مع Excel.

## تنبيه نهائي مهم

هذا الموقع مصمم لتسهيل عملية جمع بيانات الموظفين وتجهيزها للترحيل إلى ZenHR، لكنه يتعامل مع بيانات عالية الحساسية.

لذلك يجب الالتزام بثلاث قواعد أساسية:

1. لا تنشر أي ملف يحتوي بيانات موظفين على GitHub public.
2. لا تعتمد على إخفاء زر الأدمن كحماية.
3. لا تستخدم الموقع رسميًا قبل اختبار Firestore Rules والتأكد من أن الأدمن فقط يستطيع قراءة النتائج.

عند الالتزام بهذه النقاط، يمكن استخدام الموقع كحل عملي وسريع ومنظم لجمع بيانات الموظفين وترتيبها قبل عملية الترحيل.
